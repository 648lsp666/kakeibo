# Kakeibo 精简离线自动同步设计

日期：2026-07-15

状态：待用户书面审阅

## 1. 目标

为个人记账应用提供足够可靠、完全自动的多设备同步，同时严格控制系统复杂度：断网时可以继续记账，恢复联网后自动上传；多个设备看到同一账号的账单、分类和预算；日常使用不要求手动点击同步。

首版不追求完整协同编辑、版本历史或可审计冲突恢复。发生同一记录的并发修改时，以服务器最后收到的有效写入为准。

## 2. 设计原则

- **最少组件**：只使用 Supabase Auth、Postgres、Realtime 和现有 IndexedDB，不引入 Clerk 或另一套后端。
- **本地即时可用**：用户操作先在 IndexedDB 生效，不等待网络。
- **服务器为共享结果来源**：联网后由 Supabase 保存账号在所有设备间共享的最终状态。
- **自动而安静**：启动、回到前台、网络恢复和 Realtime 唤醒时自动同步；正常成功不打扰用户。
- **宁可少做**：不为低概率场景建设复杂协议。

## 3. 系统边界

### 3.1 Supabase

Supabase 同时承担身份和数据后端：

- 邮箱 OTP 登录与会话恢复。
- `transactions`、`categories`、`budgets` 三类账号业务数据。
- Row Level Security 按当前用户隔离数据。
- Realtime 仅作为“云端可能变化”的唤醒信号，客户端收到后重新拉取权威数据。

不引入 Clerk。Clerk 只能替换认证体验，不能替代账单同步；在本项目中增加它只会多出一个供应商和令牌集成边界。

### 3.2 IndexedDB

IndexedDB 保存：

- 当前账号的本地可见账本。
- 尚未获得服务器确认的 outbox 操作。
- 最近成功同步时间和少量同步元数据。

匿名数据使用 `kakeibo` 数据库；已登录账号使用 `kakeibo-user-<user-id>`。账号切换必须原子关闭旧句柄并打开目标工作区，任何较早的异步打开结果都不得覆盖后来选择的账号。

### 3.3 WebDAV 与文件导出

保留现有 JSON/WebDAV 导出作为手动灾备。它不参与自动同步，不与 Supabase 双向合并，也不承担账号认证。

## 4. 云端数据模型

三类业务记录共享以下同步字段：

- `id`：客户端生成的稳定文本 ID。
- `user_id`：当前 Supabase 用户 ID，只由认证上下文确定。
- `payload`：现有领域对象，保持金额精度、`YYYY-MM-DD` 日期和稳定分类 ID。
- `updated_at`：服务器接收并确认当前版本的时间。
- `deleted_at`：空值表示有效，非空表示软删除。
- `last_operation_id`：客户端操作 ID，用于安全重试去重。

服务端通过受控的轻量 upsert/delete 接口写入，或使用数据库约束保证 `(user_id, last_operation_id)` 的重复请求不会产生重复副作用。无需通用 revision、change log、sequence 游标、删除指纹或定时历史清理。

账单的外部交易号继续在同一用户范围内唯一，避免 CSV 重复导入。

## 5. 本地 outbox

每个本地业务修改与对应 outbox 项在同一个 IndexedDB 事务内提交。outbox 只保存同步所需的最小信息：

- `operationId`
- `entityType`
- `entityId`
- `operation`：`upsert` 或 `delete`
- `payload`
- `createdAt`
- `attemptCount`
- `nextAttemptAt`
- `lastError`（可选）

同一记录的连续待上传操作可以安全压缩：多个 upsert 只保留最终 payload；delete 覆盖此前未上传的 upsert。不同记录仍按入队顺序上传。

格式永久无效的单项进入本地隔离状态，不阻塞其他操作。网络、限流和服务器临时错误继续自动重试。

## 6. 自动同步流程

同步在以下时机触发：

- 登录会话恢复完成。
- 应用启动或回到前台。
- 浏览器/系统报告网络恢复。
- 本地产生新 outbox 项。
- Supabase Realtime 通知当前用户的数据发生变化。

一次同步循环：

1. 拉取当前账号三类云端记录。
2. 在单个本地事务中更新服务器基线。
3. 对仍有本地待上传操作的实体重新叠加本地 payload，避免用户刚做的离线修改在界面上闪回。
4. 按顺序上传可重试的 outbox。
5. 成功后应用服务器确认结果并删除对应 outbox 项。
6. 若本轮有上传，再短拉取一次，获得其他设备可能同时写入的最终状态。

同一账号同一记录发生冲突时，服务器最后收到的有效操作成为最终状态。客户端不展示冲突对话框，也不保存被覆盖版本。

## 7. 删除

删除同步为软删除：服务器设置 `deleted_at`，正常查询和本地可见列表排除该记录。首版不提供“最近删除”或 30 天版本恢复。

普通 upsert 不应意外复活已经软删除的记录；如果用户未来需要恢复功能，应新增显式 restore 操作，而不是恢复本设计已删除的复杂历史系统。

## 8. 认证、首次登录与账号切换

- 使用 Supabase 邮箱 OTP，不引入密码管理或 Clerk。
- 会话令牌由 Supabase SDK 保存，不进入业务数据库、JSON 导出或 WebDAV 备份。
- 首次登录前先生成一份匿名账本 JSON 备份。
- 用户确认后，将匿名账本写入目标账号工作区并进入 outbox；稳定 ID 保持不变。
- 若云端已存在相同 ID，以服务器最后确认的写入规则处理，不做字段级合并。
- 登出或切换账号时，如仍有待上传操作，提示数量；用户确认后保留该账号本地数据库，停止其同步循环，再切换工作区。
- 每个同步循环绑定启动时的用户 ID 和工作区句柄。中途切换账号时，旧循环必须取消或丢弃结果，禁止跨账号写入或上传。

## 9. 状态与错误处理

用户只看到五类状态：

- 已同步
- 同步中
- 离线 · N 项待同步
- 需要重新登录
- 同步失败

网络与服务器临时错误使用带抖动的指数退避；恢复联网或回到前台会立即重试。认证错误暂停网络同步但保留本地修改。协议或数据错误只隔离对应 outbox 项，并在设置页提供简短说明和重试入口。

正常同步成功保持安静，不弹 toast。设置页保留“立即重试”和手动灾备入口，但日常同步不依赖用户操作。

## 10. 测试与验收

自动化测试至少覆盖：

- 匿名与不同账号的 IndexedDB 严格隔离，包括重叠异步切换。
- 本地业务写入与 outbox 的原子提交和异常回滚。
- 断网新增、修改、删除后界面立即更新。
- 网络恢复后 outbox 自动清空，重复请求不产生重复副作用。
- 拉取服务器数据时，本地 pending payload 仍保持可见。
- 两设备先后修改同一记录时，服务器最后收到的写入最终出现在两端。
- 软删除不会被普通后台拉取或旧 upsert 意外复活。
- Realtime 只触发拉取，不直接信任通知 payload。
- 会话过期、重新登录、登出和账号切换不会丢 outbox 或串账本。
- 首次登录前备份与匿名账本迁移可重复恢复。

发布前进行一次真实双设备验收：设备 A 离线记账，设备 B 在线修改其他记录，A 恢复网络后两端最终一致。

## 11. 明确不做

首版不实现：

- Clerk 或其他额外身份服务。
- 通用 revision 冲突协议。
- `change_log`、sequence 增量游标与完整快照回退。
- 30 天覆盖历史、最近删除恢复和永久删除指纹。
- pg_cron 历史清理。
- 多标签页 leader election；重复同步循环只需依靠 operation ID 保持安全。
- 端到端加密、字段级合并、协作编辑或后台常驻同步。
