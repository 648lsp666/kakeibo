# Kakeibo 全自动多设备同步设计

日期：2026-07-15

## 1. 目标

为 Kakeibo 增加个人账号下的全自动多设备同步，同时保留现有 PWA 的离线可用性。用户在任意设备记账、删除账单、维护自定义分类或预算后，界面必须立即响应；联网时变更在秒级上传，其他已打开设备通过实时通知更新。应用关闭、离线或实时消息丢失后，重新打开、恢复网络或回到前台时必须自动补齐。

“全自动”在本设计中的边界是：应用打开时实时同步，应用重新打开或回到前台时立即追平；不承诺 PWA 完全关闭后持续执行后台任务。

## 2. 已批准的产品决策

- 使用托管云后端与个人登录，不继续以 WebDAV 作为主同步协议。
- 每个账号拥有独立账本，不支持家庭共享账本。
- 完整离线可写，联网后自动补同步。
- 同记录并发修改自动解决，不要求用户逐条选择；被覆盖版本保留 30 天。
- 删除使用 30 天墓碑，过期离线写入不能复活已删除记录。
- 同步账单、删除状态、自定义分类和预算规则；主题、会话令牌与设备偏好留在本机。
- 首版不做端到端加密。使用 HTTPS、托管数据库静态加密与行级权限隔离。
- 使用邮箱一次性验证码登录。
- 本地写入后秒级推送，并订阅其他设备的实时变更。
- 首次登录安全合并本地与云端数据，并在迁移前生成本地 JSON 备份。
- 平时静默重试；只有连续失败、身份过期或需要用户处理时才显著提醒。
- 优先使用免费额度，个人使用超出额度后再升级。

## 3. 方案选择

采用 Supabase Postgres、Auth、Row Level Security 和 Realtime，配合现有 IndexedDB 构建本地优先同步层。

未选择的方案：

- Firebase/Firestore：原生离线与实时能力强，但会与现有 IndexedDB 形成双缓存，需要大幅重写当前数据访问层，并引入不同的查询和计费模型。
- Cloudflare D1 + Workers：数据库免费额度充足，但认证、实时通知、变更游标和同步 RPC 均需自行建设，个人项目的运维与实现成本更高。
- 纯 WebDAV：适合完整文件备份，不适合可靠的记录级实时同步、并发冲突和删除传播。

WebDAV 保留为手动灾备导入导出，不参与日常自动同步。

## 4. 总体架构

系统分为四个边界清晰的单元。

### 4.1 本地仓储

IndexedDB 继续作为 UI 的唯一读写入口。页面与 hooks 不直接读写 Supabase，避免网络状态渗透到业务组件。

本地数据库按账号命名空间隔离：现有 `kakeibo` 数据库视为匿名本地账本；登录后使用由用户 ID 派生的独立数据库。首次迁移只把匿名账本复制到用户数据库一次。退出或切换账号不会复用另一个账号的本地数据库。

数据库升级到 v2，新增：

- `outbox`：尚未被服务器确认的本地 mutation。
- `sync_meta`：设备 ID、远端游标、首次迁移状态与最后成功同步时间。
- `budgets`：把目前保存在 `sync_config` JSON 中的预算规则迁移为一条规则一条记录。

现有 `transactions` 和 `categories` 增加本地同步元数据。系统分类仍由应用种子数据维护，不进入云端；仅自定义分类参与同步。

### 4.2 领域写入服务

所有新增、修改与删除必须通过领域仓储完成。同一个 IndexedDB 事务中同时：

1. 更新业务记录；
2. 写入唯一 `mutation_id` 的 outbox 事件；
3. 更新本地同步状态；
4. 触发现有 Zustand 刷新信号。

UI 不等待网络即可成功完成本地操作。写入 outbox 失败时，业务记录也必须回滚，避免出现永远无法上传的本地变更。

### 4.3 SyncEngine

`SyncEngine` 是独立于 React 页面生命周期的单实例服务，职责包括：

- 维护登录会话与当前用户同步命名空间；
- 在登录、启动、网络恢复、回到前台和 Realtime 通知后触发增量拉取；
- 串行推送 outbox，实施指数退避与随机抖动；
- 把远端变更原子地应用到 IndexedDB；
- 发布 `idle`、`syncing`、`offline`、`pending`、`auth-required`、`error` 状态；
- 防止同一 mutation 被并发推送。

多标签页使用 Web Locks 选举一个主动同步器；不支持 Web Locks 时允许多个同步器运行，但服务器端的 mutation 幂等约束必须保证结果一致。BroadcastChannel 用于把同步结果与状态通知其他标签页。

### 4.4 云端权威存储

Supabase 保存账号范围内的当前记录、删除墓碑、mutation 收据和变更日志。客户端只持有公开 anon key；所有业务访问由用户 JWT 和 RLS 约束。service-role key 不得进入前端包。

## 5. 云端数据模型

### 5.1 业务表

`transactions`、`categories`、`budgets` 共用以下同步字段：

- `id`：客户端生成的稳定 ID。
- `user_id`：Supabase Auth 用户 ID。
- `revision`：该记录的服务器递增版本。
- `updated_at`：服务器确认时间。
- `deleted_at`：空值表示有效记录，非空表示删除墓碑。
- `last_mutation_id`：最近一次产生当前版本的 mutation。
- `last_device_id`：提交当前版本的设备。

账单保留现有金额精度、日期含义、来源、外部交易号与分类 ID。`(user_id, external_id)` 在外部交易号存在时建立唯一约束，继续保证 CSV 导入去重。

### 5.2 幂等与历史表

`applied_mutations` 以 `(user_id, mutation_id)` 唯一，记录服务器已接受的操作及返回 revision。网络超时后的重复提交返回同一结果，不重复写业务数据。

`change_log` 使用单调递增 `sequence`，保存实体类型、实体 ID、旧值、新值、操作类型、设备、mutation 和服务器时间。客户端使用 sequence 作为增量游标。

可恢复历史与包含完整内容的墓碑保留 30 天。过期后每日清理任务移除旧值、新值和业务表墓碑，但把 `(user_id, entity_type, entity_id)` 写入紧凑的 `deletion_registry`。该注册表不保存金额、备注或分类内容，并长期保留，用于拒绝长期离线设备对已永久删除 ID 的普通 upsert。永久删除后的重新创建必须生成新 ID。

服务端同时公布当前最小可用 sequence。设备游标早于该值时不得继续增量应用，而是下载当前账号完整快照、原子替换服务器基线，再重新叠加本地 outbox。这样离线超过 30 天的设备仍能自动恢复，且不会依赖已清理的日志。

### 5.3 权限

所有表启用 RLS：

- 已登录用户只能读取 `user_id = auth.uid()` 的行。
- 用户只能创建归属于自己的行，不能修改 `user_id`。
- 客户端不直接改 revision、服务器时间或历史表。
- mutation 通过受控数据库函数提交，函数再次校验 `auth.uid()`。

## 6. Mutation 与同步协议

本地 outbox 事件包含：

- `mutation_id`
- `device_id`
- `entity_type`
- `entity_id`
- `operation`：`upsert` 或 `delete`
- `base_revision`
- `payload`
- `created_at`
- `attempt_count`

一次同步按以下顺序执行：

1. 使用本地 sequence 游标拉取全部远端遗漏变更，直到分页结束；游标已过期时改用完整快照。
2. 在单个本地事务中应用远端权威基线并推进游标。若某实体还有本地 outbox，则在基线上重新叠加这些待提交 mutation，保持用户当前看到的本地意图；原始 `base_revision` 不被偷偷改写，最终仍由服务器按冲突规则裁决。
3. 按 outbox 创建顺序分批调用 `apply_mutations` RPC。
4. RPC 在一个数据库事务中校验身份、去重 mutation、解决冲突、更新业务表、写 change log 并返回结果。
5. 客户端应用服务器返回的权威版本，再删除已确认 outbox 项。
6. 如果推送产生了新的服务器 sequence，再执行一次短拉取，确保本地游标追平。

Realtime 不直接携带权威业务状态，只充当“有新 sequence”的唤醒信号。连接中断或通知丢失不会破坏最终一致性。

## 7. 冲突与删除规则

- mutation 的 `base_revision` 与服务器当前 revision 相同：正常应用。
- revision 已过期：后到服务器的 mutation 成为当前版本，覆盖前一版本，并把覆盖关系写入 change log。
- 已删除记录收到基于删除前 revision 的过期 upsert：拒绝复活，返回当前墓碑。
- 完整墓碑过期后，`deletion_registry` 继续拒绝相同 ID 的普通 upsert。
- 恢复删除必须使用显式 `restore` 行为并基于墓碑当前 revision；首版 UI 仅在“最近删除”中提供恢复。
- 两个设备新增相同外部交易号：唯一约束只接受一条，另一条映射到已存在记录并从 outbox 移除。
- 删除自定义分类不会删除历史账单；账单继续保留分类 ID，UI 使用“已删除分类”回退标签。
- 系统分类不上传，稳定 ID 由应用版本保证一致。

“后到服务器者生效”避免依赖不可信的设备时钟。长期离线设备可能最后覆盖同一条记录，但被覆盖版本可在 30 天内恢复。当前交易缺少编辑入口，因此实际冲突主要集中在分类与预算，概率较低。

## 8. 认证与设备

- 登录使用 Supabase 邮箱 OTP。
- 首次登录成功后生成并持久化随机 `device_id`；清空站点数据视为新设备。
- 会话令牌由 Supabase 客户端本地保存，不进入业务同步、JSON 导出或 WebDAV 备份。
- 身份过期时暂停推送但保留 outbox，显示“需要重新登录”；重新登录后继续同步。
- 登出前必须提示本机仍有多少未同步变更。用户确认登出后保留该账号的本地数据库，但关闭其 SyncEngine。切换账号时打开对应用户数据库，禁止把前一账号的数据读入或上传给新账号。

首版使用 Supabase 内置邮件能力满足个人使用。若开放给更多用户，再配置自有 SMTP、发送域名与邮件可达性监控。

## 9. 首次启用与迁移

首次登录执行一次受保护的迁移流程：

1. 导出当前 IndexedDB 的完整 JSON 快照并在本机保存迁移标记。
2. 拉取账号的云端完整快照。
3. 云端为空时，把本地账单、自定义分类和预算转换为 outbox mutation。
4. 两端都有数据时，按稳定 ID 合并；账单额外按外部交易号去重。
5. ID 相同但内容不同的记录按普通冲突规则处理，并保留历史。
6. 同步成功且本地游标追平后，写入 `migration_complete`。

迁移中断可安全重试。mutation 幂等表确保重复执行不会重复导入。迁移前的本地 JSON 快照不会自动上传，其中也不包含登录令牌或 WebDAV 密码。

## 10. 用户界面

### 10.1 登录与引导

未登录时应用仍可作为纯本地账本使用。设置页提供“开启多设备同步”，输入邮箱后发送验证码。登录成功后展示首次合并说明与本地备份确认，随后自动开始迁移。

### 10.2 同步状态

导航或设置入口使用低干扰状态点，设置页展示完整状态：

- `已同步 · 刚刚`
- `同步中…`
- `离线 · 待同步 N 项`
- `同步受阻 · 点击处理`
- `需要重新登录`

普通短暂失败不弹窗。连续失败达到阈值、权限错误、迁移失败或身份失效才显示持久通知。用户始终可以点击“立即重试”。

### 10.3 历史与恢复

设置页新增“最近删除与冲突”入口，展示 30 天内被覆盖或删除的记录。恢复会生成新的 mutation，而不是直接修改本地数据库。

WebDAV 区域改名为“手动灾备”，保留上传、下载与 JSON 导出能力，并明确它不会参与自动同步。

## 11. 错误处理与可观测性

- 网络离线：立即切换为 offline，不消耗重试次数；网络恢复后同步。
- 5xx、超时和限流：指数退避并加入随机抖动，最大间隔五分钟；回到前台可提前重试。
- 401：刷新会话一次，仍失败则进入 auth-required。
- 403、RLS 或协议错误：停止相关 mutation，保留 outbox 并显示可处理错误。
- 单条非法数据：隔离为 dead-letter，不阻塞其他 mutation；设置页允许导出诊断信息与重试。
- Realtime 断开：状态可降级，但定期增量拉取和前台补偿继续保证一致性。
- 客户端只记录 mutation ID、实体类型、状态码和时延，不把金额、备注、邮箱或令牌写入日志。

同步健康指标包括最后成功时间、待同步数量、最老待同步时长、连续失败次数、当前游标与 Realtime 连接状态。

## 12. 安全边界

- 所有生产流量必须使用 HTTPS。
- 前端仅包含 Supabase URL 与 anon key；service-role key 只存在于受控运维环境。
- 所有公开表启用并测试 RLS。
- RPC 不信任客户端传入的 user ID、revision 或服务器时间。
- 登录重定向只允许已配置的生产与本地开发地址。
- JSON/WebDAV 灾备不包含 Supabase 会话令牌。
- 首版不做端到端加密；服务端管理员理论上可以读取业务数据，此限制必须在同步设置说明中明确。

## 13. 测试策略

### 13.1 单元测试

- outbox 与业务数据的原子写入。
- mutation 序列化、重试、幂等确认与 dead-letter。
- 增量 change log 应用与游标推进。
- revision 冲突、墓碑阻止复活和恢复 mutation。
- sync 状态机、退避、身份过期与前台重试。
- 首次迁移、外部交易号去重和预算数据迁移。

### 13.2 数据库集成测试

- 每张表的 RLS 隔离和越权拒绝。
- `apply_mutations` 的事务原子性、幂等性和并发 revision。
- change log sequence 严格递增、允许空洞，并正确处理分页与过期游标的完整快照回退。
- 30 天可恢复墓碑、历史清理与长期删除指纹防复活。
- 两个账号数据完全隔离。

### 13.3 浏览器 E2E

使用两个独立浏览器上下文代表两台设备，验证：

- A 新增后 B 秒级出现。
- A 离线连续记账，重连后完整上传。
- Realtime 消息丢失后，B 回到前台仍能补齐。
- 两端并发修改同一预算，结果确定且历史可恢复。
- 一端删除后，另一端过期写入不能复活记录。
- 登录过期时 outbox 不丢失，重新登录后继续。
- 首次登录的空云端、非空云端、中断重试与重复导入。
- 多标签页不会造成重复记录。

### 13.4 回归要求

现有记账、CSV 导入、分类、预算、统计、WebDAV 灾备、PWA 离线启动与全部 UI 测试必须继续通过。金额精度、日期语义、稳定分类 ID 和外部交易号去重不得改变。

## 14. 发布与回退

按以下阶段交付：

1. 建立 Supabase 项目、SQL migration、RLS 与本地测试环境。
2. 升级 IndexedDB v2，并在不开启云同步时保持现有本地行为。
3. 引入领域写入服务与 outbox，先在单设备假服务器下验证。
4. 接入 Auth、RPC、增量拉取和 Realtime。
5. 完成首次迁移、同步状态与恢复界面。
6. 使用测试账号进行双设备灰度，最后默认开放同步入口。

云同步使用功能开关。关闭开关只暂停云端通信，不删除本地或云端数据。出现严重问题时可回退到纯本地模式，并使用迁移前 JSON 或 WebDAV 灾备恢复。

## 15. 成功标准

- 在线设备上的普通变更在 5 秒内出现在另一台在线设备。
- 离线产生的变更在恢复网络后无需手动操作即可上传。
- 重复请求、标签页并发与同步重启不产生重复账单。
- 删除不会被过期离线设备复活。
- Realtime 丢失不影响最终一致性。
- 设备离线超过日志保留期后能够自动完整重建，并保留本地待同步变更。
- 同步失败不会阻塞记账或丢失 outbox。
- 两个账号无法通过客户端 API 互相读取或修改数据。
- 首次迁移可中断重试，并保留可下载的迁移前备份。
- 所有既有自动化测试与新增双设备测试通过。
